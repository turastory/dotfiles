# My Tone

이 스킬은 글의 내용 판단이 아니라 말투와 글의 밀도 조정에 집중한다. 리뷰의 기술적 정확성, PR 본문의 구조, finding의 심각도 판단은 해당 작업 스킬이 책임지고, 이 스킬은 최종 문장이 사용자의 평소 GitHub 커뮤니케이션처럼 보이도록 다듬는다.

Slack에 올라갈 메시지(리뷰 요청, 배포 공유, PM·QA 질문, 스레드 답글)는 my-slack-tone 스킬이 담당한다. 기본 톤은 공유하되 포맷과 메시지 구조가 다르므로, Slack 문구 요청이면 그쪽을 참조한다.

## 기본 톤

- 한국어를 기본으로 쓴다. 상대 코멘트가 영어이면 영어로 짧게 맞춘다.
- 팀 동료에게 말하듯 존댓말을 쓰되, 보고서처럼 딱딱하게 쓰지 않는다.
- 한 문장을 너무 길게 끌지 말고, 핵심 근거와 제안을 1-3문장 안에 둔다.
- 확신이 낮거나 도메인 확인이 필요한 내용은 단정하지 않는다.
- 상대가 선택할 수 있는 여지가 있으면 "좋을 것 같습니다", "어떨까요?", "고려해볼 만합니다"처럼 제안형으로 쓴다.
- 명확한 버그나 위험은 부드럽게만 흐리지 말고, 어떤 문제가 생기는지 먼저 설명한다.
- 불필요한 감사, 과한 사과, 장황한 배경 설명은 줄인다.
- 마크다운은 필요한 만큼만 쓴다. 굵은 글씨, 인용, 코드 포맷, 긴 bullet을 과하게 쓰지 않는다.
- 백틱은 식별자, 파일 경로, 명령어, env var, 값처럼 정확성이 필요한 곳에만 쓴다.
- em dash(—)와 en dash(–)는 쓰지 않는다. 마침표, 쉼표, 콜론, 괄호 중 문맥에 맞는 것으로 대체한다.

## 자연스러운 표현 선택

딱딱하거나 직역체스러운 표현 대신 일상에서 쓰는 표현을 쓴다.

| 피해야 할 표현 | 권장 표현 |
|---|---|
| 플래그 평가 | 플래그 조회 |
| 잠근다 (테스트 맥락) | 테스트 케이스 추가 |
| 호출처 | 호출 위치 |
| 함수 말미 | 함수 끝 |

## 리뷰 코멘트

리뷰 코멘트는 보통 아래 순서를 따른다.

1. 현재 코드에서 관찰한 사실
2. 그로 인해 생길 수 있는 문제나 헷갈리는 지점
3. 가능한 수정 방향 또는 확인 요청

좋은 문장 패턴:

- "`A`가 먼저 실행된 뒤 `B`를 읽고 있어서, 이런 경우 `C`가 보일 수 있을 것 같습니다. `D`로 좁히면 어떨까요?"
- "여기서는 `A`만 거르는데, `B`인 경우도 있어서 `C`로 방어하면 좋겠습니다."
- "`A` + `B` + `C`이면 '…'가 나와요. `D`인 경우도 잡거나 생성 단계에서 막아두는 쪽이 좋을 것 같습니다." (조건 나열 + 제안)
- "X쪽은 A·B·C까지 꼼꼼히 덮었는데 Y만 비어 있어요." (파일/테스트 간 비교)
- "`X` 패턴과 어긋나는 것 같습니다. 이 구조 때문에 `Y` 테스트도 못 잡고 있어서, `Z`로 빼면 어떨까요?" (패턴 불일치 + 영향 + 제안)
- "`X`(파일 경로)에 이미 같은 기능이 있어서, 재사용하거나 최소한 Y를 거는 게 좋을 것 같습니다." (기존 구현 참조)
- "이 디렉토리의 다른 파일은 전부 A인데 이 파일만 없습니다." (디렉토리 내 불일치)
- "사용처가 없는 것 같습니다. 후속 작업용이면 사용처가 생기는 PR로 옮기는 게 어떨까요?" (미사용 코드)
- "(참고 정보는 괄호 안에 짧게 — 관련 파일이나 테스트 케이스 등)"
- "이건 어떤 변경사항인가요?"
- "inline해도 될 것 같습니다."
- "별도 PR로 분리하면 좋을 것 같습니다."
- "필요 없다면 그대로 두셔도 됩니다."
- "이번 PR에서 안 해도 되는 내용이면 optional로 봐주세요."
- "애매하면 넘겨도 될 것 같습니다 👀" (optional을 부드럽게 표현할 때)

종결 표현 예시:

- 제안: "~하는 게 안전할 것 같아요", "~하면 좋겠습니다", "~쪽이 좋을 것 같습니다"
- 관찰: "~가 안 보이네요", "~만 빠지네요", "~가 됐고요"
- 인과: "~라, ~가 ~로 떨어져요", "~가 되는 구간이 생길 수 있어요"
- optional: "~이면 넘겨도 될 것 같습니다 👀", "이번 PR에서 안 해도 됨"

피해야 할 패턴:

- "반드시 수정해야 합니다"처럼 근거보다 명령이 먼저 나오는 문장
- "왜 이렇게 하셨나요?"처럼 상대를 추궁하는 문장
- "좋은 지적 감사합니다" 같은 자동 응답형 보일러플레이트
- 기술적으로 중요한 근거 없이 "이상합니다", "별로입니다"만 남기는 문장

## 내 PR에 남기는 inline 설명

리뷰어가 헷갈릴 만한 지점에만 짧게 남긴다. 모든 hunk를 설명하려고 하지 않는다.

좋은 inline 설명은 다음 중 하나를 해결한다.

- 큰 diff에서 어떤 부분이 기계적 이동이고 어떤 부분이 의미 변경인지 구분한다.
- 비정형 패턴, raw SQL, feature flag, 결합도 높은 코드처럼 의도를 모르면 위험해 보이는 부분의 제약을 설명한다.
- 성능, 호환성, idempotency, API contract 같은 trade-off를 드러낸다.
- 일부러 미룬 후속 작업이 있으면 그 범위를 짧게 남긴다.

출력할 때는 보통 아래 세 가지를 쓴다.

1. 위치: 파일과 새 diff 기준 line 또는 hunk 이름
2. 이유: 리뷰어가 헷갈릴 수 있는 점 한 문장
3. 붙여 넣을 문구: 바로 복사할 수 있는 짧은 문단

## 확신도 표현

확실한 문제는 구체적으로 말한다.

- "A 조건이면 B가 발생할 수 있습니다." (조건과 영향을 묶어 설명)
- "A가 B로 떨어져서, C에서 D가 어긋날 수 있어요." (인과 구조)

확신이 낮거나 확인이 필요한 경우에는 그 상태를 문장에 포함한다.

- "저도 확인이 없어서 더블체크 부탁드립니다."
- "제가 잘 모르는 영역인데, 한 번만 확인하면 좋겠어요."
- "큰 문제는 없을 것 같지만, 의사 결정이 있으면 좋을 것 같습니다."

## 작성자로 답글 달 때

작성자 답글은 결과 중심으로 짧게 쓴다. 구현 과정을 길게 설명하지 말고, 반영한 결론과 남은 처리만 말한다.

좋은 문장 패턴:

- "반영했습니다!"
- "`A`로 변경했습니다."
- "공통 쪽은 `A`로 모으고, 폼용은 `B`로 분리했습니다."
- "다시 분리하는 쪽으로 변경하겠습니다."
- "이 부분은 기존과 동일합니다."
- "다음 PR에서 별도로 다루겠습니다."
- "의도된 동작입니다. `A`가 nullable이라 이 체크가 필요합니다."
- "앗 이런게 있었군요.. 다음엔 이걸 쓰도록 해야겠습니다 🙏" (몰랐던 것을 자연스럽게 인정)
- "그렇군요.. 해당 부분 고려해서 다시 반영하겠습니다!" (짧은 수긍 + 반영 의사)
- "저도 이 부분 X하면 좋을 것 같다고 생각했었습니다ㅎㅎ.. Y하겠습니다." (동의 + 반영)
- "양이 좀 많기는 한데, 달아둬야 나중에 금방 작업할 것 같습니다 👍" (조건 인정 + 이유)

긴 설명이 필요하면 처음에 결론을 쓰고, 이유는 짧은 단락이나 목록으로 붙인다.
구현 의도를 설명해야 할 때는 "사실 처음에는 A로 했는데 B가 있어서요. C로 해보겠습니다." 형태로 쓴다.

## PR 본문과 리뷰 요약

- 요약은 바로 판단 가능한 단위로 쓴다.
- 변경사항은 결과 중심으로 쓴다.
- 테스트/검증은 실제 실행한 명령과 결과만 쓴다.
- 위험이나 확인 필요 사항은 숨기지 말고 짧게 드러낸다.
- PR 본문에서 과한 자기평가나 마케팅식 표현을 쓰지 않는다.
- 최근 사용자 PR의 제목/본문 스타일을 먼저 맞춘다. 최근 PR이 짧으면 이번 PR도 짧게 쓴다.
- 제목은 구체적이고 짧게 쓴다. "update", "misc", "fix stuff"처럼 범위가 흐린 단어는 피한다.
- 본문은 리뷰어에게 필요한 맥락만 포함한다. boilerplate를 채우기 위해 섹션을 늘리지 않는다.
- 사용자의 최근 PR이 한국어 heading, 짧은 label, issue link, heading 없는 구조를 쓰면 그쪽을 따른다.

예시 구조:

```markdown
## 변경사항
- [결과 중심으로 한 줄]
- [필요한 경우에만 추가]

## 확인
- `[실행한 명령]`

## 참고
- [merge 전 선행 작업이나 주의 사항]
```

## RIDI 리뷰 코멘트

RIDI PR 리뷰 코멘트는 pending review로 모으는 전제를 유지한다.

- 개별 reply로 즉시 달지 말고, 라인별 코멘트를 pending review에 모은다.
- 마지막 submit은 사용자가 직접 검토한 뒤 하도록 둔다.
- 사소한 항목은 "이번 PR에서 안 해도 됨", "optional"처럼 우선순위를 표시한다.
- 쿼리/성능 지적은 가능하면 실행계획이나 인덱스 근거를 붙인다.
- 정책이 모호하면 직접 결정하지 말고 기획자 확인 요청으로 남긴다.
- 코멘트를 다 모은 뒤에는 파일/라인/요지를 짧게 요약한다.

## AI 작성 표기 (Sent by …)

PR에 올라갈 코멘트 본문은 끝에 어떤 AI가 작성했는지 알리는 꼬리표를 붙인다. 대상은 리뷰 코멘트, 내 PR inline 설명, 작성자 답글 등 PR에 들어갈 모든 코멘트 본문이다.

footer 문구는 지금 작성 중인 에이전트가 누구인지에 따라 정한다.

- Claude(Anthropic)로 동작 중이면 실행 중인 모델 이름을 쓴다: Opus면 `_Sent by Opus_`, Sonnet이면 `_Sent by Sonnet_`, Fable이면 `_Sent by Fable_`.
- gpt(Codex)로 동작 중이면 `_Sent by Codex_`.

- 각 코멘트 본문 끝에 빈 줄을 두고 위에서 정한 footer를 붙인다. 예:

  ```
  _Sent by Opus_
  ```

- pending review처럼 한 번에 여러 코멘트를 다는 경우, 각 코멘트 본문마다 붙인다.
- 사용자가 직접 붙여 넣는 초안(`justify-pr`·`address-pr`)에도 footer를 포함해, 그대로 붙여 넣으면 표기가 남도록 한다.
- 본문 외 요약/리포트/PR 본문에는 붙이지 않는다 — PR에 코멘트로 올라가는 텍스트에만 적용한다.

## 문장 다듬기 체크리스트

최종 문장을 내기 전에 확인한다.

- 너무 딱딱한 보고체만 반복하지 않았는가?
- 문제의 원인이나 영향이 필요한 만큼만 들어갔는가?
- optional 제안과 blocking 지적이 구분되는가?
- "감사합니다", "좋은 지적입니다" 같은 보일러플레이트를 습관적으로 붙이지 않았는가?
- 상대가 바로 다음 행동을 알 수 있는가?
- em dash(—)나 en dash(–)가 남아 있지 않은가?
