# My Slack Tone

이 스킬은 Slack 표면 전용이다. 기본 말투(존댓말, 제안형, 근거 먼저, 과하지 않은 밀도)는 my-tone 스킬과 공유하므로, 내용 판단이나 문장 밀도가 애매하면 my-tone의 기본 톤 섹션을 함께 참조한다. 여기서는 Slack 특유의 포맷, 멘션, 이모지, 메시지 유형별 구조를 다룬다.

## 기본 톤 (Slack 델타)

- GitHub 코멘트보다 한 단계 가볍게 쓴다. "~부탁드립니다~", "~같아서요", "..!" 같은 부드러운 종결을 쓴다.
- 말줄임표는 ".." 두 점으로 쓴다 ("아직 셀프 리뷰 중입니다..", "지금 PR들이 너무 많은 것 같아서..").
- 화살표 `->` 는 조건과 결과, 재현 플로우, 단계 나열을 압축할 때만 쓴다 ("일반 액션 -> N / M 형태로 진행도가 보여서..", "등록 -> 수정 -> 저장", "연계 액션 중 일부만 달성 -> 아무것도 체크되지 않음"). 일반 문장의 접속에는 쓰지 않는다 ("~것 같아서 -> 1번이 맞을 것 같습니다"는 어색함. "~것 같아서 1번처럼 계속 보여주는 게 맞지 않을까 싶습니다"로 쓴다). em dash(—)는 여기서도 쓰지 않는다.
- 자기 실수나 모르는 영역은 숨기지 않고 앞에 둔다 ("제가 잘못 이해하고 있는 부분이 있는 것 같아서 싱크를 맞추고 싶은데요", "제가 장르홈쪽은 잘 몰라서, 제쪽에서 판단할 수 있는 건은 아닌 것 같아 전달드립니다").
- 데드라인이나 우선순위가 있으면 명시한다 ("내일 오전까지 마스터에 들어가야 하는 것은 이게 마지막이지 않을까 싶습니다").

## Slack 포맷 규칙

- 헤딩(#) 문법이 없으므로 섹션 구분은 `*굵게*` 한 줄로 한다 ("*PR 스택*", "*[리뷰 프로세스 정리]*").
- 굵게는 문장 안 핵심 구절 강조에 쓴다. 메시지당 1~3곳이 적당하고, 정책·조건·변경 요지처럼 상대가 놓치면 안 되는 부분에만 쓴다 ("*연계 액션이 여러 개인 미션을 등록*했을 때의 문제").
- 목록은 불릿(•) 또는 번호를 쓴다. 옵션 나열은 번호, 관찰·사실 나열은 불릿.
- PR 링크는 raw URL로 한 줄에 두고(unfurl 목적), 바로 아래 불릿으로 `[scope] 커밋 스타일 제목`을 붙인다.
- 참고성 링크(스레드, Asana, Figma)는 `<url|설명 텍스트>` 형태로 문장에 녹인다 ("<url|관련 스레드>", "<url|관련 Figma 코멘트 링크>").
- 정보성 공유는 "fyi." 프리픽스로 시작할 수 있다.
- 백틱(코드 포맷)은 `tb_event_participation.event_group_id` 처럼 정확한 네이밍이 필요하거나 `.` 같은 문자가 섞여 경계가 헷갈리는 식별자에만 쓴다. event_idx, mission_id 같은 간단한 식별자는 평문으로 쓴다.

## 멘션

- 팀 전체 리뷰 요청은 유저그룹 멘션을 첫 줄에 단독으로 둔다 (`<!subteam^S095P0PHCLC>` = product-pay-engineer).
- 특정인에게는 `@아무개님,` 으로 시작한다. 이름 뒤 "님"을 붙인다.
- 참조만 필요한 사람은 `(cc @아무개 @아무개)` 로 묶는다. 보통 첫 줄의 대상 멘션 문장 끝에 붙이고 ("@소정님, 이 이슈 한 번 확인 부탁드립니다 :bow1: (cc @규림)"), 제3자 전달처럼 맥락상 끝이 자연스러우면 마지막 줄에 둔다.
- 답글 대상이 명확한 스레드 안에서는 멘션을 생략해도 된다.

## 이모지

문장 끝 장식은 0~2개. 습관처럼 모든 문장에 붙이지 않는다.

- 부탁·감사: `:bow1:` (가장 자주), `:allo_pray:`, `:pray:`
- 긍정·동의: `:allo_thumbsup:`, `:+1:`
- 곤란·아쉬움: `:tears:`, `:cry-wc:`
- 확인 중: `:eyes:`, `:allo_hmm:`
- 상태 표시(목록에서 기능적으로 사용): `:white_check_mark:` 완료, `:merge:` 머지됨, `:ballot_box_with_check:` 리뷰 중/진행 중, `:construction:` 작업 중

## 메시지 유형별 구조

### 리뷰 요청

1. 첫 줄: 멘션 (팀그룹 또는 개인)
2. PR 링크 + 불릿 제목 (여러 개면 각각)
3. 요청 문구 + 필요한 맥락 1~3문장

맥락에 들어가는 것: PR을 분리한 이유, 배포 순서 ("백엔드 먼저 배포하기 위해 PR을 분리했습니다"), 데드라인, 남은 작업 상태 ("웹 QA쪽은 코드 정리 중인데 정리되면 공유하겠습니다!").

좋은 문장 패턴:

- "리뷰 부탁드립니다~"
- "확인 부탁드려요 :bow1:"
- "리뷰 요청드립니다! 그리 길지 않습니다 :+1:"
- "늦은 시간이지만.. 요 PR 하나만 한 번 봐주시면 감사하겠습니다 :allo_pray:"
- "간단한 PR 하나 리뷰 부탁드립니다~"

### PR 스택 / 컨텍스트 스레드

여러 PR을 묶어 공유할 때는 트래커 형태로 쓴다.

```
참여 현황 백오피스 PR / 컨텍스트 스레드

*PR 스택*
1. [backoffice] proto + backends: <url> :white_check_mark: :merge:
2. [backoffice] 프론트엔드 데이터/로직: <url> :construction:
...
2,3 PR 리뷰 요청드립니다..!
```

상태가 바뀌면 원 메시지를 수정해 이모지를 갱신하는 전제로 쓴다.

### PM·QA 질문 / 이슈 리포트

1. 첫 줄: 대상 멘션 + (cc)
2. 배경: 어떤 이슈/링크에서 출발했는지, 내가 어디까지 이해했는지
3. 요지를 명시한다 ("요지는 두 가지 입니다!")
4. 옵션이 있으면 번호로 나열하고, 각 옵션에 짧은 주석을 단다 ("1. 라벨 컬럼을 빼고 자동으로 보여줌 (*Figma 스펙과 일치*)")
5. 내 의견이 있으면 "제 생각으로는.." + 불릿으로 각 선택지의 장단점을 쓴다

좋은 문장 패턴:

- "두 가지 방향 중 하나로 가면 좋을 것 같습니다 :allo_pray:"
- "혹은 *~하는 케이스*가 일반적이지 않은 케이스일까요?"
- "이것 관련해서 정헌님이 코멘트를 주셨는데, ~라고 하시네요." (제3자 의견 전달)
- "한 가지 헷갈리는 케이스가 있어서요."
- "상황: ~ -> 이 때 ~하고 있습니다." (재현 조건을 화살표로 압축)

### 배포·진행 상황 공유

결과를 먼저 쓰고, 상대가 해야 할 액션이 있으면 명시한다.

- "수정해서 스테이징에 반영했습니다."
- "prod에 백오피스 배포되었습니다!"
- "기존에 있던 체크리스트의 경우 등록 -> 수정 -> 저장을 해서 컬럼 값을 갱신해주어야 합니다 :allo_pray:" (후속 액션)
- "UI 노출 Feature Flag ON 하겠습니다! (cc)" (실행 선언 + 관련자 멘션)
- 남은 이슈는 불릿 + 링크 + 상태 한 줄로 나열한다 ("<url|프로그레스바 간헐적으로 잘못 표시> - 재현 어려움으로 보류").

### 스레드 답글

짧게. 결론이나 상태만 말한다.

- "넵" / "감사합니다!" / "그렇군요..!"
- "네 방금 오픈 했습니다!"
- "확인했습니다! 원인 정리드리면," + 불릿 (원인 설명이 필요할 때)
- "의도된 구현은 아닙니다! ~ 한 번 확인해볼게요."
- "앗 잠시 밖에 나와있는데 들어가서 보겠습니다 :eyes:"
- "머지했습니다. 지금 배포 가능하실까요!?"

## 피해야 할 패턴

- 공지문처럼 딱딱한 개조식만 나열하는 메시지 (섹션 헤더 + 불릿만 있고 문장이 없는 형태)
- "안녕하세요", "수고 많으십니다" 같은 인사 보일러플레이트로 시작하는 메시지
- 모든 문장 끝에 이모지를 붙이는 것
- 긴 배경 설명 후에야 요청이 나오는 구조. 요청·결론을 앞쪽에 두고 배경은 뒤에 붙인다.
- GitHub 마크다운 문법(`##`, `**`, `[텍스트](url)`)을 그대로 쓰는 것. Slack은 `*굵게*`, `<url|텍스트>`.

## 전송 전 확인

- 초안은 사용자가 직접 붙여 넣는 전제로 쓰고, AI 표기 꼬리말은 붙이지 않는다 (PR 코멘트의 _Sent by AI_ 규칙은 Slack에는 적용하지 않는다).
- Slack MCP로 직접 전송할 때는 보내기 전에 최종 문구를 사용자에게 보여주고 확인받는다.
- 멘션 ID(유저그룹, 개인)는 추측하지 말고 기존 메시지나 검색으로 확인한 것만 쓴다.
